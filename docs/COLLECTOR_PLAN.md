# KIP 수집기 구현 계획

이 문서는 실제 구현 전 계획서다.  
코드, DB, API, 크롤러는 수정하지 않았다.

## 1) 목표

- 공식 또는 공개 경륜 데이터를 안정적으로 수집한다.
- 첫 단계에서는 다음 4개만 우선 수집한다.
  1. 경기 일정
  2. 출주표
  3. 선수 기본 정보
  4. 경기 결과
- 배당, 베팅 추천, 기대수익률, AI 예측은 제외한다.

## 2) 권장 수집 범위

### 1차 범위

- 경기 일정
  - 경주일
  - 시행처/경륜장
  - 회차
  - 경주번호
  - 시작 예정 시각
  - 상태
- 출주표
  - 선수번호
  - 선수명
  - 등급
  - 지역/훈련지/출신팀
  - 출전번호
  - 라인 또는 배번
  - 최근 성적 요약
  - 기어배수
  - 200m 기록
- 선수 기본 정보
  - 선수번호
  - 이름
  - 등급
  - 지역
  - 상태
  - 출신지/출신팀
  - 전법/라인 후보 정보
- 경기 결과
  - 1~3위
  - 순위
  - 경주기록
  - 기권/실격 여부
  - 확정 상태

### 후순위 범위

- 공지사항
- 선수 변경 정보
- 심판/제재 정보
- 세부 전법 통계
- 날씨/뉴스
- 배당/투표 관련 정보

## 3) 권장 처리 흐름

```text
외부 데이터
→ 수집
→ 정규화
→ 중복 판정
→ 공통 import service 또는 CSVImportService
→ DB 저장
```

### 권장 방식

- 1차는 `CSVImportService`를 최대한 재사용한다.
- 수집기는 원본 데이터를 바로 DB에 쓰지 않는다.
- 표준화된 중간 레코드를 만든 뒤, 기존 import 검증/중복/업서트 규칙을 태운다.
- 장기적으로는 CLI, 관리자 업로드, 자동 수집이 같은 import core를 공유하는 구조가 가장 좋다.

## 4) 권장 디렉터리 구조

```text
backend/app/collectors/
├── base.py
├── sources/
│   ├── kcycle.py
│   └── data_go.py
├── parsers/
├── normalizers/
├── dto.py
└── services/

scripts/
├── collect_schedule.py
├── collect_players.py
├── collect_entries.py
└── collect_results.py
```

### 역할 분리

- `sources`
  - 웹 페이지 또는 OpenAPI 호출 지점을 캡슐화한다.
- `parsers`
  - HTML, JSON, XML, 파일 다운로드를 파싱한다.
- `normalizers`
  - 외부 필드를 KIP 스키마로 표준화한다.
- `services`
  - 수집 순서, 재시도, 저장, 재검증을 조립한다.
- `scripts`
  - 수동 실행용 CLI 진입점이다.

## 5) 데이터 매핑 원칙

### 경기장

- 외부 `시행처`, `경륜장`, `장소`를 `tracks`에 매핑한다.
- `tracks.code`는 내부용 표준 코드로 관리한다.
- `tracks.name`은 사용자 표시명으로 사용한다.
- `tracks.region`은 광명/창원/부산 같은 운영 권역을 담는다.
- `tracks.address`는 확인 가능한 경우에만 채운다.

### 선수

- 외부 `선수번호`는 우선 순위가 가장 높은 외부 고유키다.
- 외부 상세 URL의 식별자도 함께 보존하는 후보가 필요하다.
- `players.player_number`는 외부 선수번호와 정합성을 유지한다.
- `players.grade`는 등급문자열을 그대로 저장한다.
- `players.region`은 출신지/훈련지/팀 정보를 어떤 규칙으로 반영할지 먼저 정해야 한다.

### 경주

- `race_date`, `track_id`, `race_number`를 기본 고유키로 쓴다.
- 외부 race ID가 있으면 보조 고유키로 저장한다.
- `scheduled_start_time`은 출주표 또는 일정 페이지의 시작 시각을 이용한다.
- 상태는 `scheduled`, `in_progress`, `completed`, `cancelled` 같은 내부 enum으로 정규화한다.

### 엔트리

- `race_id + player_id` 중복을 금지한다.
- `entry_number`는 경주 내 출전번호로 저장한다.
- `lane_number`는 페이지에서 제공될 때만 저장한다.
- `lineup_position`은 라인 또는 정렬 순서를 내부 기준으로 저장한다.

### 결과

- `finish_position`은 순위 숫자로 저장한다.
- `finish_time`은 원본 표기 그대로 또는 표준 시간 문자열로 정규화한다.
- `result_status`는 `finished`, `dnf`, `dq`, `scratched` 같은 상태로 정규화한다.
- `points`는 제공될 때만 저장한다.

## 6) 고유키 및 중복 정책

### 경기 고유키

- 기본: `경기일 + 경기장 + 경주 번호`
- 보조: 외부 race id

### 선수 고유키

- 기본: `선수번호`
- 보조: 외부 player id

### 재수집 정책

- 신규 데이터는 insert 한다.
- 동일 키인데 내용이 같으면 skip 한다.
- 동일 키인데 중요 필드가 다르면 update 한다.
- 경주 출전 명단이 바뀌면 entry를 재계산한다.
- 결과가 확정되면 result를 update 한다.
- 이미 종료된 경주의 결과는 원칙적으로 수정하되, 변경 이력은 남긴다.

## 7) 수집 일정 권장안

### 일정

- 전날 저녁
  - 다음날 경주 일정 예비 수집
- 당일 오전
  - 출주표 초안 수집
- 경기 시작 1~2시간 전
  - 출주표 갱신
  - 선수 변경 사항 반영
- 경기 직전
  - 최종 출주표 확인
- 경기 종료 후
  - 결과 수집
- 다음 날 새벽
  - 결과 재검증 및 누락 보정

### 운영 파라미터

- 요청 간 대기 시간: 기본 1~3초
- 재시도 횟수: 3회 권장
- 타임아웃:
  - 목록/일정: 10~15초
  - 출주표/결과: 20~30초
- 실패 로그:
  - 소스
  - 시각
  - 경주 키
  - 실패 단계
  - 에러 요약
- 부분 실패:
  - 한 경주 실패가 전체 배치를 멈추지 않도록 한다.
- 중복 실행:
  - 같은 날짜/회차를 다시 돌려도 idempotent 해야 한다.
- 수동 재실행:
  - 특정 날짜 또는 특정 회차만 재수집 가능해야 한다.
- 변경 감지:
  - 레코드 해시 또는 핵심 필드 비교로 변화를 판단한다.

## 8) 오류 처리 정책

- 소스 레벨 실패
  - 사이트 전체 오류나 API 불능은 배치 실패로 기록한다.
- 행 레벨 실패
  - 특정 경주나 특정 선수만 실패하면 그 행만 건너뛴다.
- 형식 오류
  - 파서가 이해할 수 없는 경우에는 원본 저장 없이 실패 처리한다.
- 제한 초과
  - 요청량이 많으면 더 느린 재시도로 전환한다.
- 인증 실패
  - API 키 문제는 즉시 중단한다.

## 9) 기존 CSVImportService와의 연결 계획

### 단기

- 수집기 출력물을 CSV 컬럼 규격으로 맞춘다.
- 그 후 기존 `CSVImportService` 또는 동일한 import validator를 호출한다.
- 이렇게 하면 다음을 재사용할 수 있다.
  - 필수 컬럼 검증
  - 중복 방지
  - dry-run
  - 행별 오류 수집
  - 관리자 업로드와 같은 저장 규칙

### 중기

- `CSVImportService`에서 컬럼 파싱만 분리해 공통 `ImportService`로 재구성한다.
- CLI, 관리자 업로드, 자동 수집이 같은 core를 공유한다.

## 12) 최종 권장안

- 현재 KIP의 첫 수집기는 `KCYCLE 공식 사이트`를 기준으로 설계하고,
- `data.go.kr OpenAPI`를 구조화 검증 및 보강용으로 붙이는 방식이 가장 현실적이다.
- 구현은 반드시 `수집 → 정규화 → 기존 import core 재사용` 순서로 가져가야 한다.
